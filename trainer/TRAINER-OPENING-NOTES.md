# Trainer Opening Notes

These notes are for the trainer only.
They are not part of the participant workshop flow.

---

## Goal of the Opening

Set the tone.
Reduce anxiety.
Make the workshop feel simple, practical, and safe.

Participants should feel:

- I do not need to code
- I do not need to understand everything
- I just need to describe what I want and react to the result

---

## Opening Script (2–3 minutes)

You can say this almost word for word:

“Welcome everyone.

Today is not a coding workshop.
You do not need to know how to write code, read code, or use developer tools in a deep way.

The goal today is simple:
to show how you can take an idea, describe it in plain language, and turn it into something real with AI.

We are going to work in a very simple pattern:

Describe → Review → Improve

That means:
you describe what you want,
you look at what comes back,
and then you ask for a change.

That is the core skill.

The point is not to build a perfect application.
The point is to experience how quickly an idea can become something concrete.

So if something looks wrong, that is fine.
If the first result is incomplete, that is fine.
That is part of the exercise.

Your job today is not to get everything right.
Your job is to guide the system.

By the end of the session, you should have something working and a better feel for how to use AI in a practical way.”

---

## Transition Into the Workshop

Then say:

“We’ll start by seeing a quick example.
After that, you’ll choose one simple idea, build a first version, and improve it step by step.”

If needed, add:

“If something breaks, don’t worry.
We’ll keep moving.
This is a workshop about confidence and interaction, not perfection.”

---

## What to Emphasize

Repeat these ideas during the session:

- “Start simple.”
- “The first version is just a starting point.”
- “Pick one small change.”
- “You do not need to fix it yourself — just describe what should change.”
- “The skill is in guiding, not coding.”

---

## What to Avoid Saying Early

Do not lead with:

- agents
- skills
- MCP
- prompt files
- repo structure
- how the system is wired together

Do not turn the opening into a technical explanation.

The opening should be about:
- confidence
- simplicity
- practical value

---

## If the Group Looks Nervous

Say:

“You are not expected to understand the implementation.
You are only expected to describe what you want and react to what you see.”

And:

“If you have ever told someone how you want a dashboard, report, or workflow to look, you already have the skill we need today.”

---

## If the Group Is More Technical Than Expected

Say:

“We are deliberately keeping the first part simple.
At the end, we can talk about what is happening behind the scenes and how this is guided.”

---

## Demo Setup Reminder

Before starting, make sure:

- the assistant/chat is open
- you have your demo prompt ready
- you know how to open the built app
- you are ready to show one small refinement after the first build

Do not improvise the first demo live if you can avoid it.

---

## Demo Prompt

Use a safe, simple example such as:

“Build me a simple KPI dashboard for a sales team. Show total revenue, deals closed, average deal size, and top five customers. Use realistic sample data. Make it clean and easy to read. It should open in a browser with no setup.”

Then refine with something like:

“Add a search box and make the layout clearer.”

---

## Closing Reminder

At the end of the workshop, reinforce:

- they started from an idea
- they got something real
- they improved it through conversation

Only after that should you optionally discuss what was happening behind the scenes.
